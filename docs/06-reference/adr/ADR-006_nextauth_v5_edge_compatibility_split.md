# ADR-006: NextAuth (v5) 設定ファイル群の Edge Runtime 対応に向けた分割設計

## ステータス
**承認済**

## コンテキストと課題 (Context & Problem Statement)

認証機能の基盤として NextAuth.js (Auth.js v5) を導入するにあたり、以下の技術的課題に直面した。
- Next.js のミドルウェア (`middleware.ts`) を用いた全ルート保護を実装するためには、読み込まれる認証設定ファイルが **Edge Runtime** 環境で動作する必要がある。
- しかし、データベースアクセスに必須となる Prisma Adapter や、独自ログイン（Credentials）におけるパスワード検証に必要な `bcryptjs` は Node.js 環境に依存しており、Edge Runtime では実行時エラーとなる。
- 設定を単一の `auth.ts` のみに記述してしまうと、この依存関係の制約によりミドルウェア自体が正常に動作しない。

## 決定事項 (Decision)

NextAuth v5 の設定を、以下の役割を持つ **2つのファイルに明確に分割するアーキテクチャ・パターン** を採用する。

1. **`src/auth.config.ts` (ミドルウェア・Edge 対応用)**
   - Node.js 環境に依存しない軽量な設定のみを定義する。
   - `pages`（ログインページのパス）、シークレット渡しだけで完結する OAuth プロバイダー、およびミドルウェアの核となる `callbacks.authorized` によるルート保護ロジックを含む。
   
2. **`src/auth.ts` (アプリケーション本体・Node環境用)**
   - 本体として `auth.config.ts` をスプレッド構文 (`...authConfig`) にて継承する。
   - Edge では動作不可能な `PrismaAdapter` と `CredentialsProvider`（`bcryptjs` のパスワード検証ロジック等）を統合する。
   - 最終的に API ルートや Server Actions 向けに各種関数群（`handlers`, `auth`等）をエクスポートする。

## 選定理由 (Rationale)

- **公式推奨プラクティス**: Auth.js (NextAuth v5) 公式ドキュメントで提示されている標準的な Edge Runtime の回避策であり、今後のパッケージのアップデートにも強い。
- **パフォーマンスとセキュリティの両立**: 毎回のページ遷移時にフックされるミドルウェア側にはDB接続等の余計なコードがロードされないため高速化に寄与する。一方で、バックエンド処理側では Prisma を用いた堅牢なデータ検証体制を安全に構築可能である。

## 代替案と却下理由 (Considered Options)

- **Edge 対応の Prisma (\@prisma/extension-accelerate) を使う案**:
  - Prisma を Edge 環境で動作させ、ファイルを1つにまとめることも技術的には可能。
  - *却下理由*: 外部サービス登録等の追加設定が必要となり、現行のローカル開発（PostgreSQL への直接接続）が複雑化し、MVP（Minimum Viable Product）実装のスピードが削がれるため見送った。
  
- **ミドルウェアを使わずに Server Component 内のみで認可チェックを行う案**:
  - Edge Runtime を一切考慮せず、各ページのコード内でセッション確認を行う。
  - *却下理由*: 開発者がページ作成時に保護設定を漏らす（セキュアなページが誰でもアクセス可能になる）ヒューマンエラーのリスクを排除できないため見送った。

## 将来への影響 (Consequences)

- **運用ルールの固定化**: 今後、新しい認証の仕組みを追加する場合、「それが Node.js 環境に依存するか否か」を開発者が論理的に判断し、適切な設定ファイルへ追加しなければならない。
- このアーキテクチャの背景はチーム開発に際して躓きやすいポイントであるため、`ARCHITECTURE.md` 等へドキュメントとして継続的に記録しておく必要がある。
