## ADR-003: メール送信プロバイダーの選定

### ステータス

承認済み

### コンテキスト

確認メール等を送信するためのサービス（プロバイダー）および発行するトークンの有効期限が未決定でした。メール送信基盤がないと、今後のPhase2におけるパスワードリセットなどの機能実装を進めることができません。

### 決定

Resendを採用し、トークンの有効期限は24時間と決定しました。（案A）

### バージョン選定理由

- **なぜこのバージョンか**: Next.js (React Server ComponentsやServer Actions) と非常に親和性が高く、`react-email` を併用することでUIコンポーネントとしてメールテンプレートを構築できるため。トークンの有効期限に関する24時間という設定は、ユーザビリティとセキュリティのバランスが良いと判断されました。
- **AIカットオフ対策**: Resendは最近普及してきたサービスであるため、APIのバージョンや仕様に注意し、公式ドキュメントに沿った実装を行う必要があります。
- **非推奨API回避**: 特になし。
- **LTS/サポート期間**: Resendは比較的新しいサービスですが、Vercelエコシステムとの連携も強固です。

### 代替技術との比較

| 技術/手法 | 採用/却下 | 理由 |
| :--- | :--- | :--- |
| SendGrid | 却下 | 機能は豊富だが、セットアップがやや煩雑でNext.jsネイティブの体験としてはResendに劣るためMVPとしてはオーバースペック。 |
| Amazon SES | 却下 | インフラ管理のコストが高く、MVPには適していないため。 |

### 影響

- **ポジティブ**: 開発体験が高く、メールテンプレートをReactコンポーネントとして実装できるためメンテナンスしやすいです。構成も容易です。
- **ネガティブ**: 大規模な配信や高度な機能が必要になった場合は、エンタープライズ向けのプロバイダー（SendGrid等）への移行を検討する必要があります。

### AIへの指示

Resendと `react-email` を使用してメール送信機能を実装してください。トークンの有効期限はシステム内で24時間として扱います。
