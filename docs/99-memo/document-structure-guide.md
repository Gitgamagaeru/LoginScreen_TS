# ドキュメント構成ガイド

この文書は、本プロジェクト（`01_Login`）で作成された各種ドキュメントの「意味」や「なぜその構成になっているのか」を説明するための補足資料です。（`AGENTS.md` のルールに基づき、`docs/99-memo/` 配下に作成しています）

## 1. ドキュメントの全体構造（7文書）

本プロジェクトのドキュメントは、「企画（What）」から「運用（How）」に至るまで、上から下へと段階的に読めるように設計されています。

全体の目次・牽引役として `docs/MASTER.md` が存在し、以下の構成で並んでいます。

| フォルダ | ファイル名 | 何を決めるファイルか（意味・役割） |
|---|---|---|
| `00-planning` | **PROJECT.md** | **「作るもの」と「作らないもの」の定義**<br>誰に向けて何を作るのか、必須機能（MVP）は何か、パフォーマンスやセキュリティはどうするかという「要件」をまとめます。 |
| `01-context` | **DOMAIN.md** | **「言葉」と「ルール」の定義**<br>開発者同士（またはAIと人間）で言葉の認識がズレないよう、「User」「Session」といった用語の意味や、パスワード管理などの「ビジネスルール」を固定します。 |
| `02-design` | **ARCHITECTURE.md** | **「技術」と「全体構造」の定義**<br>「どの技術（Next.js, Prisma, PostgreSQL）を使うのか」「全体がどう繋がっているか」という、技術的な根幹を定めます。 |
| `03-implementation` | **PATTERNS.md** | **「コードの書き方」の定義**<br>実際のプログラミングにおいて、どのような命名規則を使うか、NextAuthのセッションはどう取得すべきかなど、コード品質を保つルールを記載します。 |
| `04-quality` | **TESTING.md** | **「品質担保」の定義**<br>作ったものが「正しく動いているか」をどうやって確認するか（今回であれば手動テストの観点など）を定めます。 |
| `05-operations` | **DEPLOYMENT.md** | **「公開・運用」の定義**<br>完成したものをVercelにどうやってアップロードするか、環境変数には何が必要かといった、本番環境への道筋を記載します。 |

## 2. なぜこの構成になっているのか？（設計意図）

1. **AI駆動開発（AI-Driven Development）に最適化するため**
   - 全てを1つのファイルに書くと、AIの読み込み制限（コンテキストウィンドウ）を圧迫してしまいます。役割ごとにファイルを分割することで、例えば「コードを書くときは `PATTERNS.md` だけを読む」といった効率的な制御が可能になります。
2. **手戻りを最小限にするため**
   - `PROJECT.md` で合意を取らずにいきなりコードを書き始めると、「実は2段階認証も必要だった」等で大きく修正が発生するリスクがあります。左の数字（00〜05）の順に決めていくことで、設計のブレを防ぎます。

## 3. その他のファイルについて

* **`.github/ISSUE_TEMPLATE/`**
  * `task.md`, `bug_report.md`: GitHubでタスク管理を行うための「入力フォーム（ひな形）」です。「目的」や「完了条件」が見出しとして固定されるため、作業内容が明確になります。
* **`AGENTS.md`**
  * プロジェクトに参加する「AIエージェント（私のようなチャットAIやCopilot）」に対して、絶対守るべきルール（ブランチの切り方や、ドキュメントの保存先など）を強制するための設定ファイルです。

## 4. このガイドの読み方
今後、新しい機能（例：パスワードリセット機能）を作るときは、以下の手順でドキュメントを更新していきます。

1. `PROJECT.md` の Phase2 に着手として追記
2. `DOMAIN.md` に「VerificationToken」のルールを追記
3. 実際にコードを書きながら、新しいパターンを見つけたら `PATTERNS.md` を更新

## 5. src/ ディレクトリの構成（コードの保存場所）

設計ドキュメントに対応する「実際のソースコード」が保存される `src/` 配下の構成とその意味です。本プロジェクトは **Next.js (App Router)** を採用しているため、以下のルールで構成されています。

| フォルダ / ファイル | 意味・役割 |
|---|---|
| `src/app/` | **画面（ルーティング）の中心**<br>フォルダを作って中に `page.tsx` を置くと、それがそのままWebページのURLになります。<br>・例：`app/page.tsx` 👉 `http://localhost:3000/` （トップ・ダッシュボード）<br>・例：`app/login/page.tsx` 👉 `http://localhost:3000/login` （ログイン画面） |
| `src/app/layout.tsx` | **全ページ共通のレイアウト枠**<br>すべての画面を包み込むHTMLの大枠（`<head>` や `<body>`、共通メタタグなど）を定義します。 |
| `src/app/globals.css` | **全体共通のデザイン設定（CSS）**<br>Tailwind CSSを使っており、ベースカラーやグラスモーフィズム等の大元となる設定を記述しています。 |
| `src/components/` *(今後作成)*| **再利用可能なUI部品や論理コンポーネント**<br>「ボタン」や「入力フォーム」、共通ヘッダーなど、複数の画面で使い回す部品を `app/` から切り出して配置します。 |

## 6. プロジェクト直下の主な設定ファイル

`src/` フォルダと同じ並び（トップディレクトリ）に存在する、多数の設定ファイル（`.json` や `.mjs` 等）の意味です。多くは自動生成され、最初は初期設定のままで動作します。

| ファイル名 | 意味・役割 |
|---|---|
| `package.json` | **アプリの設計図と部品リスト**<br>プロジェクトの名前や、インストールされている外部ライブラリ（Next.js, Tailwindなど）のバージョン一覧、そして `npm run dev` 等のコマンドが定義されています。 |
| `package-lock.json` | **詳細なバージョンの固定記録**<br>`package.json` で指定したライブラリがさらに裏で使っている全ての部品のバージョンを厳密に固定・記録します。（人間が直接編集することはありません） |
| `eslint.config.mjs` | **Linter（コードの品質チェッカー）の設定**<br>コードの書き間違いや「推奨されない書き方」を自動検知するツール「ESLint」の設定ファイルです。※Next.js 15から採用された新しいFlat Config形式（`.mjs`）です。 |
| `tsconfig.json` | **TypeScript の設定**<br>TypeScript（型付きJavaScript）をどのように解釈・コンパイルするか、型チェックの厳密さをどうするかを設定するファイルです。 |
| `next.config.ts` | **Next.js 本体の大元設定**<br>外部画像ドメイン（Googleのプロフィール画像等）の許可や、ビルドの挙動、独自環境変数のルールなど、フレームワーク自体の根幹動作をカスタマイズします。 |
| `postcss.config.mjs` | **CSS変換ツールの設定**<br>Tailwind CSS などの記述を、古いブラウザでも理解できる標準的なCSSに変換・最適化するツール「PostCSS」の設定です。 |
| `.gitignore` | **Gitにアップロードしないファイルの指定**<br>秘密情報（`.env` 等）や、自動生成される重いファイル（`.next/` や `node_modules/`）など、GitHubに送ってはいけないものを指定します。 |
