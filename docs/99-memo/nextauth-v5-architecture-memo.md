# NextAuth.js (Auth.js v5) アーキテクチャ解説メモ

NextAuth v5 (Auth.js) における認証設定ファイルの役割と、設定を2つのファイル (`auth.config.ts` と `auth.ts`) に分割した技術的な背景についての解説です。

## 1. なぜファイルを分割するのか？ (Edge Runtime 問題)

Next.js の「ミドルウェア (`middleware.ts`)」は、ページへのアクセス要求が来るたびに最速で実行される必要があるため、**Edge Runtime** という非常に軽量で制限された環境で動作します。

一方で、データベースの接続に使う **Prisma** や、パスワードの暗号化/照合に使う **bcryptjs** といったライブラリは、Node.js のコア機能に深く依存しているため、Edge Runtime では正常に動作しません。

もしすべての設定を一つのファイル (`auth.ts`) にまとめてしまい、それを認証ガード用のミドルウェアからインポートしようとすると、「Edge 環境では実行できないライブラリが含まれている」としてビルドや実行時にエラーになってしまいます。

これを完全に回避しつつ、ミドルウェアの恩恵を受けるために、**「ミドルウェアでも動く軽量な設定（DB不要）」** と **「ログイン処理の実体（DB必須）」** を分離するアプローチが標準となっています。

## 2. 各ファイルの役割とコード解説

### ① `src/auth.config.ts` (ミドルウェア向け・軽量設定)

このファイルは、Edge Runtime で動くミドルウェアから読み込まれることを前提としています。そのため、Prisma や bcrypt といった Node.js 依存のコードは含めません。

**主な役割:**
*   **`pages`**: カスタムログインページ (`/login`) などのルーティング先を指定します。
*   **`providers` (OAuth関連)**: Google や GitHub といったソーシャルログインの設定は「クライアントIDとシークレットを渡すだけ」の軽量な設定であるため、ここに記述することが可能です。
*   **`callbacks.authorized`**: ミドルウェアの核となるロジックです。ユーザーがログインしているかどうか (`!!auth?.user`) と、アクセスしようとしているURL (`nextUrl.pathname`) に応じて、以下の判定を行います。
    *   **ダッシュボードへのアクセス時**: 未ログインなら `false` を返し、自動的に `signIn` ページにリダイレクトさせます。
    *   **ログイン済みの場合**: トップページやログイン画面にアクセスした際、すでにログインしているためダッシュボードへ強制リダイレクト（`Response.redirect`）させます。

### ② `src/auth.ts` (アプリケーション本体向け・DB接続)

このファイルは、認証用APIルートや Server Actions など、標準の Node.js 環境で動くバックエンドから読み込まれる「本体」です。

**主な役割:**
*   **継承 (`...authConfig`)**: 軽量設定である `auth.config.ts` を読み込み、その内容を展開して引き継ぎます。
*   **`adapter`**: `PrismaAdapter(prisma)` を登録し、ユーザー情報やセッション情報をデータベース (PostgreSQL) と同期させます。
*   **`providers` の統合 (Credentials)**: 
    *   `...authConfig.providers` で設定済みの OAuth プロバイダーを展開します。
    *   さらに、メールアドレスとパスワードを用いる `CredentialsProvider` をここに追加定義します。
    *   この中の `authorize` 関数は、ユーザーの入力情報をもとに **DBからユーザーを検索 (`prisma.user.findUnique`)** し、**パスワードを検証 (`bcrypt.compare`)** するという「重い処理」を行うため、必ずこちらのファイルに配置する必要があります。
*   **エクスポート**: APIルートやコンポーネントで呼び出す `handlers` (GET/POST処理用) や `auth` (セッション取得用)、`signIn`, `signOut` といった関数群を生成し公開します。

## 3. 全体のデータフロー

1. ユーザーがブラウザで `/dashboard` など保護されたページにアクセスを試みます。
2. **ミドルウェア** がページ表示より先に起動し、`auth.config.ts` の `authorized` コールバックを実行してチェックします。未ログインなら `/login` へ弾かれます。
3. ユーザーがログイン画面 (`/login`) でメールとパスワードを入力し送信します。
4. **`auth.ts`** に定義された `CredentialsProvider` が走り、Prisma と bcrypt を使って検証を行います。成功するとセッションが発行されます。

この「2段構えのアーキテクチャ」により、**Next.js の表示の高速性（エッジ・ミドルウェア）を活かしながら、Prisma を使った堅牢で複雑な認証システム（Node.js ランタイム）を両立させる**ことができます。
